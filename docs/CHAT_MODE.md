# Chat Mode

Chat mode запускает Streamlit UI, где можно писать QA-задачи обычным языком, загружать требования и продолжать работу с уже созданной сессией.

## Запуск

Самый простой запуск:

Windows PowerShell:

```bash
.\run_chat_windows.bat
```

Windows cmd:

```bat
run_chat_windows.bat
```

macOS/Linux:

```bash
chmod +x run_chat.sh
./run_chat.sh
```

Launcher создаёт `.venv`, ставит зависимости, создаёт локальный конфиг и предлагает сохранить `OPENAI_API_KEY` в `.env`.

Если нажать Enter и не вводить ключ в терминале, запуск не остановится: приложение откроется, а ключ можно будет добавить через браузер в блоке `Настройка AI`.

Для принудительной переустановки зависимостей:

Windows PowerShell:

```powershell
.\run_chat_windows.bat --reinstall
```

macOS/Linux:

```bash
./run_chat.sh --reinstall
```

## Как добавить OPENAI_API_KEY

Рекомендуемый способ для новичка — через браузер:

1. Запустите `.\run_chat_windows.bat` в PowerShell, `run_chat_windows.bat` в cmd или `./run_chat.sh` на macOS/Linux.
2. Откройте UI.
3. Вставьте ключ в поле `OPENAI_API_KEY`.
4. Включите `Сохранить ключ в локальный .env`, если не хотите вводить ключ при каждом запуске.
5. Нажмите `Сохранить ключ`.

Через `.env` в корне проекта:

```env
OPENAI_API_KEY=sk-...
```

Через переменную окружения:

Windows:

```bash
set OPENAI_API_KEY=sk-...
```

macOS/Linux:

```bash
export OPENAI_API_KEY=sk-...
```

Замените `sk-...` на реальный ключ. Файл `.env` не коммитится, ключ не показывается в UI после сохранения и не пишется в логи.

Ручной запуск:

```bash
ai-qahelper chat
```

Или отдельной командой:

```bash
ai-qahelper-chat
```

Если console scripts не попали в `PATH`, используйте:

```bash
python -m ai_qahelper.cli chat
```

## Project Setup и Sidebar

Основной контекст задаётся во вкладке `Generate` в блоке `Project Setup`:

- `Файлы требований`: `.md`, `.txt`, `.pdf`, `.docx`, `.xlsx`, `.xls`;
- `Ссылка на тестируемый сайт` для стенда или сайта;
- `Figma file key`, если нужен макет через API;
- `Session ID`, чтобы продолжить существующую сессию.

Sidebar оставлен коротким и содержит только рабочие настройки:

- быстрый `Session ID`;
- `Тип результата`: `testcases` или `checklist`;
- информационный блок о coverage-first генерации: агент сам определяет количество проверок и создаёт coverage report;
- `Дополнительные настройки`;
- `Google Sheets export`.

Сложные настройки Site Discovery спрятаны в `Расширенные настройки Site Discovery`.

## Вкладки UI

Chat mode содержит вкладки `Dashboard`, `Generate`, `Coverage`, `Requirements`, `Test Cases`, `Checklist`, `Review` и `Export`. Текущий чат, Project Setup и запуск AI находятся во вкладке `Generate`; coverage metrics вынесены в отдельную вкладку `Coverage`; `Quality`, `Bug Reports` и настройки шаблонов доступны внутри `Review`/`Export`, чтобы верхняя навигация не перегружалась.

## Стоимость и LLM-вызовы

Открытие вкладок, просмотр таблиц, фильтрация, редактирование, сохранение JSON, утверждение final-файлов, экспорт XLSX и локальная проверка качества работают без LLM. Вкладки `Dashboard`, `Test Cases`, `Checklist`, `Quality`, `Review`, `Export` и `Settings` читают файлы текущей сессии локально и не запускают AI автоматически.

LLM вызывается только после явного действия пользователя: кнопки workflow во вкладке `Generate`, кнопок с пометкой `через AI` или отправки ручной команды. Локальная проверка качества бесплатная; `Глубокий AI-анализ качества`, `Найти серые зоны через AI` и `Предложить улучшения через AI` расходуют LLM-токены.

## Настраиваемые шаблоны документации

В `Settings` → `Templates` можно выбрать колонки для `Test Cases`, `Checklist` и `Bug Reports` без ручного редактирования YAML. Дополнительные колонки включаются и выключаются checkbox-ами; порядок можно задать числом `Порядок`. Обязательные колонки отображаются включёнными и недоступны для отключения.

Базовый шаблон QAHelper остаётся шаблоном по умолчанию. Пользовательский шаблон сохраняется в `templates/user_templates/<artifact_type>_template.json`; если задан активный `session_id`, шаблон можно сохранить в `runs/<session_id>/template_settings.json`. При загрузке используется приоритет: session template, затем user template, затем default template.

Активный шаблон добавляется в prompt генерации: агент получает список enabled-колонок, их порядок и required-поля, не должен добавлять лишние поля и должен помечать gaps через `notes` или `status = Needs review`, если такие колонки включены. Export XLSX/CSV использует только enabled-колонки и их labels. Quality Gate проверяет required-поля активного шаблона и не ругается на отключённые optional-поля.

## Agent Memory и контекст сессии

Chat mode хранит контекст текущего диалога в `st.session_state` и, когда есть `session_id`, сохраняет его в `runs/<session_id>/agent_context.json`.

Агент запоминает:

- последние загруженные requirements;
- `target_url`;
- `session_id`;
- последний созданный артефакт;
- тип документации: `testcases` или `checklist`;
- последние действия пользователя;
- предложенные следующие шаги.

Это позволяет писать короткие продолжения вроде `продолжай`, `теперь баги`, `теперь negative cases` без повторного указания всех параметров.

## Natural Language Planner

Агент понимает обычные фразы и строит план действий: ingest требований, site discovery, генерация test cases/checklists, bug drafts, подготовка или запуск автотестов, синхронизация с Google Sheets.

Если LLM planner недоступен или вернул некорректный JSON, chat-agent переключается на базовое распознавание команд по ключевым словам и показывает предупреждение.

Опасные действия требуют confirmation: запуск автотестов и операции, которые могут менять внешние Google Sheets.

Если данных не хватает, агент задаёт уточняющий вопрос. Например, для `сделай тест-кейсы` без requirements, target URL и session id он предложит загрузить требования, вставить URL сайта для Site Discovery или продолжить существующую сессию.

## Как общаться с AI QAHelper обычным языком

Пишите цель так, как сказали бы QA Lead:

```text
Вот требования, сделай smoke test cases.
```

```text
Теперь сделай negative cases.
```

```text
Создай баг-репорты.
```

```text
Покажи quality report.
```

```text
Подготовь автотесты, но не запускай.
```

После успешного действия агент показывает, что сделано, какие файлы созданы, какой `session_id` активен и что можно сделать дальше.

## QA Workflow

Во вкладке `Generate` основной запуск AI находится в блоке `QA Workflow`. Вместо формы с параметрами используются карточки готовых сценариев:

- `Generate test cases`;
- `Generate checklist`;
- `Negative coverage`;
- `Smoke coverage`;
- `Coverage report`;
- `Quality review`.

Пользователь не задаёт количество test cases или checklist items. Агент сначала анализирует требования и test conditions, затем создаёт столько проверок, сколько нужно для покрытия позитивных, негативных, граничных, UI/API/state сценариев, если они применимы. После генерации сохраняется `coverage-report.json`.

Внутри `Project Setup` и `QA Workflow` показывается готовность к запуску: найден ли `OPENAI_API_KEY`, загружены ли требования, указан ли URL стенда и активна ли `Session ID`.

Перед запуском проверяются базовые условия:

- задан валидный `OPENAI_API_KEY`;
- есть входной контекст: загруженные requirements, URL тестируемого стенда или существующая `Session ID`.

Если ключа нет, UI показывает warning `Добавьте OPENAI_API_KEY в блоке «Настройка AI», чтобы использовать AI.` и не вызывает LLM.

Если нет требований, URL тестируемого стенда и `Session ID`, UI показывает warning `Недостаточно входных данных. Загрузите требования, укажите URL тестируемого стенда или выберите существующую Session ID.` и не вызывает LLM.

Заполнение Project Setup и просмотр вкладок не вызывает AI. LLM вызывается только после нажатия кнопки карточки workflow или отправки ручной команды.

Warnings из workflow не записываются в историю чата, поэтому повторные попытки с неготовым контекстом не засоряют диалог. Когда запуск разрешён, сформированная команда отправляется в тот же chat flow и использует текущие requirements, target URL, `Session ID` и Agent Memory.

## Ручная команда

Под блоком `QA Workflow` остаётся ручной чат. Он нужен для произвольных команд, например `сделай negative cases только для авторизации`.

Отправка сообщения в ручной чат вызывает LLM и добавляет команду в историю диалога. Кнопка `Очистить чат` рядом с заголовком `Ручная команда` очищает только сообщения и pending-план. Загруженные requirements, `target_url`, `session_id`, Agent Memory и `OPENAI_API_KEY` остаются на месте.

Для созданных артефактов UI показывает путь и кнопку скачивания. Markdown-файлы отображаются как preview, JSON-файлы показывают короткую сводку, а Excel/CSV доступны для скачивания без полноценного табличного viewer.

Основные кнопки скачивания:

- `Скачать test-cases.xlsx`;
- `Скачать checklist.xlsx`;
- `Скачать coverage report`;
- `Скачать quality report`;
- `Скачать exploratory report`;
- `Скачать JSON`.

## Понятные ошибки

Chat mode показывает пользовательские сообщения вместо технических traceback:

- `Не найден OPENAI_API_KEY` — вставьте ключ в блоке `Настройка AI` в браузере, сохраните его в `.env` или задайте переменную окружения;
- `Не загружены требования` — загрузите файл требований или укажите ссылку на документ;
- `Не указан Target URL` — вставьте ссылку на тестируемый сайт;
- `Не удалось прочитать файл` — проверьте путь и доступ к файлу;
- `Не удалось запустить Playwright` — установите браузеры `python -m playwright install` или выключите Playwright;
- `Не удалось выгрузить в Google Sheets` — проверьте ссылки и `GOOGLE_SERVICE_ACCOUNT_JSON`.

## Примеры

Есть требования:

```text
Вот requirements.docx. Сделай профессиональные тест-кейсы без дублей.
```

Нет требований, есть сайт:

```text
Требований нет. Вот сайт https://example.com. Пройди до 5 страниц и сделай smoke + negative тест-кейсы.
```

Сложная команда:

```text
Посмотри требования, найди риски, сначала сделай smoke, потом negative cases, затем подготовь черновики багов.
```

Продолжение по session id:

```text
Продолжи сессию 20260426-example-com-ci-smoke и сделай checklist.
```
