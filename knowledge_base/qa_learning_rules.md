# QA Learning Rules

1. Не использовать слово "Sprint" в названиях проектов, тест-кейсов, чек-листов, баг-репортов, отчётов и выводов агента.
2. Если исходный материал содержит учебное название со словом "Sprint", заменять его на нейтральное описание функциональности.
3. Один тест-кейс должен проверять одну основную идею.
4. Не создавать дублирующие проверки.
5. Не использовать фразы:
   - работает корректно
   - отображается корректно
   - проверить функциональность
   - всё работает
6. Для текстовых полей с ограничениями всегда проверять:
   - пустое значение
   - значение меньше минимума
   - минимальное валидное значение
   - максимальное валидное значение
   - значение больше максимума
7. Если требования неполные — добавлять вопрос, gap или risk, а не выдумывать поведение.
8. Статусы Passed, Failed, Blocked не проставлять заранее, если тест ещё не был выполнен.
9. Ожидаемый результат должен быть конкретным и наблюдаемым.
10. Шаги должны быть воспроизводимыми.
11. Для API проверять status code, response body, schema, required fields, authorization, invalid data и error messages.
12. Для mobile проверять разные устройства, ориентацию экрана, слабый интернет, отсутствие интернета, восстановление сети, системные кнопки, разрешения и поведение после сворачивания приложения.
